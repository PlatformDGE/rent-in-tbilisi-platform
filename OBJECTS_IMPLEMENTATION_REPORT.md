# Отчёт о реализации модуля «Объекты»

## Результат

Полноценный frontend-модуль «Объекты» реализован в ветке `rebuild` в соответствии с `OBJECTS_IMPLEMENTATION_PLAN.md`. Общий layout, дизайн-токены, backend и внешние интеграции не изменялись.

## Реализовано

### Реестр

- поиск по внутреннему ID, адресу, кадастровому номеру, собственнику и телефону;
- фильтры по статусу, району, источнику, агенту, типу недвижимости и валюте;
- сортировка по обновлению, цене, адресу и внутреннему ID в обоих направлениях;
- переключение таблицы и карточек с сохранением предпочтения;
- обязательные поля реестра: статус, агент, источник, цена, район, адрес и дата обновления;
- отдельные empty state для пустого реестра и пустого результата фильтрации.

### Форма

Реализованы создание и редактирование со всеми заявленными полями. Добавлена проверка обязательных значений, чисел, URL, соотношения комнат и спален, этажа и этажности, а также уникальности внутреннего ID. Ошибки сохранения не очищают форму.

### Карточка

Добавлены вкладки «Обзор», «Собственник», «Медиа», «Публикации», «Показы», «Договоры» и «История». Вкладка сохраняется в URL. Для ещё не подключённых хранилищ и модулей показываются честные empty state.

### Действия

- редактирование;
- назначение агента;
- проверка дублей по внутреннему ID и ссылке источника;
- отправка на фотосъёмку;
- подготовка публикации;
- запуск сценария договора;
- архивирование.

Действия, связанные с будущими внешними модулями, меняют только локальный рабочий статус и не имитируют отправку или создание данных во внешней системе.

### История

История создаётся service-слоем и хранится вместе с объектом. Автоматически записываются создание и изменения:

- цены или валюты;
- статуса;
- имени или телефона собственника;
- ответственного агента.

Каждая запись содержит старое и новое значение, время, действие и локального автора.

### Хранение и ошибки

- добавлен интерфейс repository;
- реализован версионированный `LocalStoragePropertyRepository`;
- начальное хранилище пустое;
- добавлены пользовательские ошибки чтения и записи;
- подключён существующий глобальный Error Boundary;
- Telegram, Google Sheets и backend не подключались.

### Адаптация

Модульные стили поддерживают desktop и mobile: адаптивные формы, карточки, таблица с безопасным скроллом, прокручиваемые действия и вкладки, мобильная панель сохранения. Глобальная дизайн-система не изменена.

## Маршруты

- `/properties/:propertyId`
- `/properties/:propertyId/edit`

## Проверки качества

- строгая TypeScript-типизация проверяется командой сборки;
- все отображаемые кнопки имеют обработчики или маршруты;
- начальные и демонстрационные объекты отсутствуют;
- изменения в `server` отсутствуют;
- commit и push не выполнялись.

Финальные результаты `npm run build` и `git diff --check` приведены в сообщении о завершении задачи.

## Обновление структуры для Post Generator

Форма и domain-модель приведены к структуре актуального Post Generator `platformd.ge/post-generator.html`:

- добавлены тип сделки и условные поля аренды/продажи;
- добавлены эксклюзивность, описание, метро и Google Maps;
- расширены справочники типа недвижимости, спален, здания, состояния и дизайна;
- добавлены ванные, отопление, политика питомцев и полный набор удобств;
- добавлены депозит, ценовые диапазоны и срок аренды;
- расширены данные агента, рекрута и собственника;
- добавлен `MediaItem` с локальным preview, порядком, главным фото, признаком оригинала и placeholder для watermark;
- добавлены placeholders Telegram publication metadata без подключения Telegram;
- добавлена функция `buildPublicationDraft(property)`, возвращающая типизированные блоки сделки, локации, объекта, удобств, цены, агента, собственника, медиа и системных данных;
- обзор карточки перестроен в том же порядке, что и Post Generator.

Repository переведён на schema version 2. При отсутствии v2 он читает старый ключ v1, мигрирует существующие записи и сохраняет результат под новым ключом. Старый ключ не удаляется. История, технический ID, даты, цена, адрес, район, собственник и агент сохраняются при миграции.

Условная валидация требует состояние объекта для продажи, проверяет максимальное количество жильцов для аренды и исключает нерелевантные поля при построении публикационного draft.
