# План реализации модуля «Объекты»

## 1. Цель и ограничения

Реализовать полноценный frontend-модуль управления объектами недвижимости в ветке `rebuild` без изменений общего layout и существующей дизайн-системы.

Ограничения этапа:

- данные хранятся только в `localStorage` через repository-слой;
- backend, Telegram, Google Sheets и парсеры не подключаются;
- готовые или похожие на реальные объекты не добавляются;
- начальное состояние реестра пустое;
- тестовый объект создаётся только вручную пользователем;
- все подписи, сообщения об ошибках и empty state выполняются на русском языке.

## 2. Архитектура

### Domain

В `src/domain` будут добавлены строго типизированные модели:

- `Property` — текущее состояние объекта;
- `PropertyStatus` — допустимые статусы;
- `PropertySource`, `PropertyType`, `Currency` — справочные типы;
- `PropertyHistoryEntry` — запись истории;
- `PropertyHistoryField` — отслеживаемые поля: цена, статус, собственник и агент;
- типы входных данных для создания и редактирования объекта.

Для объекта будет использоваться технический UUID, отдельный от вводимого пользователем внутреннего ID. Даты будут храниться в ISO 8601, деньги — числом без форматированного текста.

### Repository

В `src/repositories` будет определён интерфейс `PropertyRepository`:

- получить список объектов;
- получить объект по техническому ID;
- создать объект;
- обновить объект;
- архивировать объект;
- проверить внутренний ID и ссылку источника на возможные дубли.

Реализация `LocalStoragePropertyRepository` будет:

- изолировать чтение и запись `localStorage` от React-компонентов;
- проверять и нормализовать сохранённую структуру;
- возвращать понятные ошибки при повреждённых данных или невозможности записи;
- записывать изменения объекта и историю атомарно в одной сохранённой структуре;
- поддерживать версию схемы хранения для последующих миграций.

### Services

Сервис объектов будет отвечать за сценарии:

- создание и редактирование;
- смену статуса;
- назначение ответственного агента;
- изменение собственника и цены;
- архивирование;
- проверку дублей;
- формирование записей истории.

История не будет формироваться UI-компонентами. Сервис сравнит предыдущее и новое состояние и добавит отдельную запись для каждого изменения цены, статуса, собственника или агента.

### Module

В `src/modules/properties` планируется структура:

```text
properties/
  components/
  hooks/
  pages/
  property.constants.ts
  property.formatters.ts
  property.validation.ts
  PropertiesPage.tsx
```

Компоненты реестра, формы и карточки объекта будут разделены, чтобы страницы не содержали repository-логику.

## 3. Маршруты

Будут добавлены маршруты внутри существующего layout:

- `/properties` — реестр объектов;
- `/properties/new` — добавление объекта;
- `/properties/:propertyId` — карточка объекта;
- `/properties/:propertyId/edit` — редактирование объекта.

Карточка объекта будет поддерживать вкладки через URL-параметр или вложенный маршрут, чтобы выбранная вкладка сохранялась после обновления страницы.

## 4. Реестр объектов

### Возможности

- поиск по внутреннему ID, адресу, кадастровому номеру, имени и телефону собственника;
- фильтры по статусу, району, источнику, агенту, типу недвижимости и валюте;
- сортировка по дате обновления, цене, адресу и внутреннему ID;
- направление сортировки по возрастанию и убыванию;
- переключение «Таблица / Карточки»;
- сброс активных фильтров;
- счётчик найденных объектов без имитации аналитики.

### Отображаемые данные

- статус;
- ответственный агент;
- источник;
- цена и валюта;
- район;
- адрес;
- дата обновления.

Выбранный вид и параметры фильтрации будут храниться в состоянии страницы. При пустом repository отображается empty state с действующей кнопкой «Добавить объект».

На мобильном экране таблица будет заменяться компактным карточным представлением либо получит безопасный горизонтальный скролл для обязательных колонок. Фильтры будут открываться в адаптивной панели.

## 5. Добавление и редактирование

Единая типизированная форма будет использоваться для создания и редактирования объекта.

Поля:

- внутренний ID;
- источник;
- ссылка на источник;
- адрес;
- район;
- тип недвижимости;
- комнаты;
- спальни;
- площадь;
- этаж;
- этажность;
- цена;
- валюта;
- кадастровый номер;
- имя собственника;
- телефон собственника;
- ответственный агент;
- комментарий;
- статус.

### Валидация

- обязательные поля будут явно отмечены;
- числовые значения не смогут быть отрицательными;
- спальни не смогут превышать количество комнат при заполнении обоих полей;
- этаж не сможет превышать этажность;
- ссылка источника будет проверяться как URL, если заполнена;
- внутренний ID должен быть уникальным;
- пробелы в строковых значениях будут нормализованы;
- ошибки будут отображаться рядом с полями и в общей области формы;
- при ошибке сохранения введённые данные не будут потеряны.

После успешного создания пользователь перейдёт в карточку созданного объекта. Отмена вернёт в реестр без сохранения.

## 6. Карточка объекта

Шапка карточки покажет внутренний ID, адрес, статус, цену, ответственного агента и доступные действия.

Вкладки:

1. **Обзор** — основные характеристики, адрес, источник, комментарий и служебные даты.
2. **Собственник** — имя, телефон и связанные действия редактирования.
3. **Медиа** — корректный empty state; загрузка файлов не имитируется до отдельного решения о хранении.
4. **Публикации** — empty state и локальные записи, если они появятся через действие «Создать публикацию».
5. **Показы** — empty state без вымышленных показов.
6. **Договоры** — empty state и переход к сценарию создания договора без backend-интеграции.
7. **История** — хронологический список отслеживаемых изменений.

Несуществующий `propertyId` приведёт к понятному экрану «Объект не найден» с возвратом в реестр.

## 7. Статусы

Единый справочник статусов:

1. Новый
2. Проверка
3. Закреплён
4. Нужна фотосъёмка
5. Фото готовы
6. Готов к публикации
7. Опубликован
8. Обновление
9. Показ
10. Переговоры
11. Договор
12. Сдан
13. Продан
14. Архив
15. Отклонён

Справочник будет использоваться формой, фильтрами, бейджами и действиями. Цвета статусов будут построены на существующих дизайн-токенах или аккуратных семантических дополнениях без изменения общего визуального языка.

## 8. Действия

Все отображаемые действия должны иметь завершённый локальный сценарий:

- **Редактировать** — открывает форму редактирования;
- **Закрепить за агентом** — открывает выбор агента и сохраняет назначение;
- **Проверить дубли** — проверяет внутренний ID и ссылку источника, показывает результат;
- **Отправить на фотосъёмку** — меняет статус на «Нужна фотосъёмка» после подтверждения;
- **Создать публикацию** — создаёт локальную черновую запись либо показывает явно обозначенный локальный сценарий без отправки во внешние сервисы;
- **Создать договор** — открывает существующий раздел договоров с контекстом объекта либо локальную форму-заглушку, которая не обещает сохранение до реализации contracts repository;
- **Перенести в архив** — после подтверждения меняет статус на «Архив».

Перед реализацией действий «Создать публикацию» и «Создать договор» будет выбран минимальный честный UX, который не создаёт фиктивных внешних результатов. Неактивные действия будут иметь объяснение причины, а не выглядеть рабочими кнопками.

## 9. История изменений

Запись истории будет содержать:

- уникальный ID;
- технический ID объекта;
- тип изменённого поля;
- предыдущее значение;
- новое значение;
- дату и время;
- источник действия (`Создание`, `Редактирование`, конкретное быстрое действие);
- локального автора как `Пользователь`, пока авторизация не реализована.

При создании объекта будет записано событие создания. При обновлении сервис отдельно сравнит:

- цену вместе с валютой;
- статус;
- имя и телефон собственника;
- ответственного агента.

Сохранение без изменения отслеживаемых значений не создаст пустых событий.

## 10. Состояния и обработка ошибок

Для реестра, формы и карточки будут предусмотрены:

- загрузка;
- пустое состояние;
- объект не найден;
- повреждённые локальные данные;
- ошибка чтения `localStorage`;
- ошибка записи из-за ограничений браузера;
- подтверждение опасных действий;
- восстановление интерфейса через повторную загрузку данных.

Repository-ошибки будут преобразовываться в русские пользовательские сообщения. Технические детали не будут показываться в интерфейсе.

## 11. Мобильная адаптация и доступность

- форма перестраивается в одну колонку;
- действия карточки помещаются в адаптивное меню или переносятся строками;
- вкладки доступны через горизонтальный скролл без обрезания;
- фильтры не перекрывают нижнюю навигацию;
- интерактивные элементы имеют достаточную область нажатия;
- формы имеют связанные `label`, сообщения ошибок и фокус на первой ошибке;
- диалоги доступны с клавиатуры;
- статусы не различаются только цветом.

## 12. Порядок реализации

1. Создать domain-типы, справочники и форматтеры.
2. Определить repository-контракт и версионированную localStorage-реализацию.
3. Реализовать сервис операций и генерацию истории.
4. Добавить контекст или hooks для загрузки и обновления объектов.
5. Реализовать реестр, поиск, фильтры, сортировку и два вида отображения.
6. Реализовать форму создания и редактирования с валидацией.
7. Реализовать карточку объекта и семь вкладок.
8. Реализовать все действия и подтверждения.
9. Проверить историю всех обязательных изменений.
10. Выполнить адаптивную и accessibility-проверку.
11. Проверить отсутствие demo-данных и изменений backend.
12. Запустить финальные проверки.

## 13. Критерии приёмки

- пустой реестр не содержит заранее созданных объектов;
- пользователь может вручную создать объект и увидеть его в реестре;
- поиск, каждый фильтр и каждая сортировка дают предсказуемый результат;
- таблица и карточки отображают обязательные поля;
- объект открывается, редактируется и сохраняется после перезагрузки страницы;
- все семь вкладок доступны;
- каждое изменение цены, статуса, собственника и агента появляется в истории;
- все заявленные действия работают либо честно объясняют ограничение текущего этапа;
- ошибки localStorage не приводят к белому экрану;
- mobile и desktop варианты не ломают общий layout;
- TypeScript проходит в строгом режиме;
- `npm run build` завершается успешно;
- `git diff --check` не находит ошибок;
- `git diff -- server` остаётся пустым.

## 14. Проверки после реализации

```bash
npm run build
git diff --check
git status --short
git diff -- server
```

Дополнительно будет проведена ручная проверка полного сценария: пустой реестр → создание → поиск → фильтрация → открытие карточки → редактирование обязательных полей истории → быстрые действия → архивирование → перезагрузка страницы.
